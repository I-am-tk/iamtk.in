Every blog eventually becomes an interface. A reader lands on a page, builds a mental map, and decides whether the next paragraph is worth their attention. The job of spacing is not decoration; it is pacing. Good spacing tells the reader where a thought starts, where it deepens, and where it is safe to pause.

This post is a real article, but it also exercises the MDX elements used by this site: **strong emphasis**, _quiet emphasis_, `inline code`, links, lists, tables, task checkboxes, code blocks, images, footnotes, keyboard hints, and a few mathematical notes rendered with KaTeX.

## Start with rhythm, not density

Readable posts need enough structure to scan without turning into documentation soup. I like to start with three constraints:

- Keep the measure narrow enough that long paragraphs do not become work.
- Give headings more space above than below, because they introduce a new idea.
- Treat code and tables as interruptions that deserve their own breathing room.

Those constraints sound small, but they prevent the common failure mode where every element has the same margin and the article becomes visually flat.

### Paragraphs should feel connected

Adjacent paragraphs should read like a conversation. The gap between them should be visible, but not so large that every sentence feels like a separate card. When the text contains a phrase like <mark>watch the vertical rhythm</mark>, the highlight should be obvious without hijacking the line.

The same rule applies to links. A sentence can point to the [MDX documentation](https://mdxjs.com/) without forcing the link to dominate the paragraph. A useful link is discoverable, not loud.

> The best default spacing system is one you stop noticing after the first minute of reading.

That quote is doing real work: it separates a principle from the surrounding prose without needing a special callout component.

## A tiny model for spacing

You can think about vertical spacing as a cost function. If a section has `n` ideas and each idea has a visual separation cost `s`, the reader's scan cost can be sketched as:

$$
C_{scan} = \sum_{i=1}^{n} (w_i \times s_i)
$$

That equation should render as a display equation. Inline math should be tested too: $line\ height \approx font\ size \times 1.6$.

### Code examples should be calm

A technical blog needs code blocks that can hold attention without becoming the whole page. Here is a small TypeScript helper that keeps design tokens explicit:

```ts
type RhythmStep = "tight" | "normal" | "loose";

const verticalRhythm: Record<RhythmStep, string> = {
    tight: "0.75rem",
    normal: "1.25rem",
    loose: "2rem",
};

export function spaceFor(step: RhythmStep) {
    return verticalRhythm[step];
}
```

The inline form matters too. A reader should be able to notice `spaceFor("normal")` in a sentence without feeling like the code is shouting.

## GFM is where blog posts get practical

GitHub Flavored Markdown gives posts a few useful working surfaces. Task lists are especially good for publishing checklists that are meant to be read, not edited:

- [x] Check the base paragraph measure.
- [x] Verify code blocks do not collide with surrounding text.
- [x] Add KaTeX rendering for display and inline math.
- [ ] Decide whether footnotes should be visually quieter.

Tables are just as important. They need enough spacing to scan on desktop and enough horizontal overflow behavior to survive mobile.

| Element | Purpose | Spacing note |
| --- | --- | --- |
| Heading | Starts a new idea | More margin above than below |
| Paragraph | Carries the argument | Moderate vertical rhythm |
| Code block | Pauses the prose | Larger top and bottom margins |
| Table | Compares facts | Scroll horizontally when needed |

GFM also gives us strikethrough. A sentence like ~~just install a typography plugin and call it done~~ is a useful reminder that defaults should match the site, not the other way around.

## Small typographic details

Sometimes the details are literal. A shortcut like <kbd>Ctrl</kbd> + <kbd>K</kbd> should sit comfortably inside a paragraph. A unit such as H<sub>2</sub>O or a scale like 10<sup>3</sup> should not disturb line height. A marked phrase should still feel like text, not a button.

### Nested lists should still scan

Nested content is where spacing often breaks. This ordered list should remain readable:

1. Start with the reader's path through the page.
2. Check each repeated element against that path.
3. Tune in this order:
    - body copy
    - headings
    - code blocks
    - tables and media

#### When to add a component

Add a custom MDX component when the content has a repeated semantic pattern, not merely because one article needs a special box.

##### When to keep plain Markdown

Plain Markdown is better when the structure is already known: paragraphs, lists, quotes, links, and tables should not need bespoke wrappers.

###### A final small rule

If a reader has to think about the spacing, the spacing is probably asking for too much attention.
